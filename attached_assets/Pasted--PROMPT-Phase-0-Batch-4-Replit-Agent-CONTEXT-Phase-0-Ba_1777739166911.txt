11. apps/pos-api/src/lib/jwt.ts:
    - Import { SignJWT, jwtVerify } from 'jose'
    - Import { JwtPayloadSchema, SessionJwtPayloadSchema, TerminalJwtPayloadSchema, SESSION_JWT_TTL_SECONDS, TERMINAL_JWT_TTL_SECONDS } from '@nuatis/pos-shared'
    - signSessionJwt({ tenant_id, user_id, role }) → returns signed JWT string + expires_at ISO date
    - signTerminalJwt({ tenant_id, location_id, staff_id }) → same shape
    - verifyJwt(token) → returns parsed-and-validated payload (SessionJwt | TerminalJwt) OR throws on invalid/expired
    - Algorithm pinned to HS256
    - Secret loaded from env.POS_JWT_SECRET (TextEncoder to Uint8Array for jose)

12. apps/pos-api/src/lib/passwords.ts:
    - hashPassword(plain) → bcrypt with cost 12
    - verifyPassword(plain, hash) → bcrypt compare
    - hashPin(pin) → bcrypt with cost 10 (PINs are short, balance latency)
    - verifyPin(pin, hash) → bcrypt compare

13. apps/pos-api/src/middleware/auth.ts:
    - requireAuth() factory — returns Express middleware
    - Reads `Authorization: Bearer <token>` header
    - Verifies JWT, attaches parsed payload to req.auth (typed)
    - Optional argument: required token kinds, e.g. requireAuth({ kinds: ['session'] }) or requireAuth({ kinds: ['session', 'terminal'] })
    - 401 with { error: { code: 'unauthorized', message: '...' } } on missing/invalid/expired/wrong-kind
    - Type augmentation in apps/pos-api/src/types/express.d.ts to declare req.auth shape

14. apps/pos-api/src/routes/auth.ts:
    - Express Router
    - POST /v1/auth/sign-in:
      - Validates body via SignInRequestSchema
      - Looks up staff_member by email (Supabase service_role query)
      - Checks role is 'owner' or 'manager' (cashiers MUST use PIN endpoint, not password)
      - bcrypt-verifies password against staff.password_hash (NEW COLUMN — see migration step 16)
      - Issues session JWT
      - Returns SignInResponseSchema-shaped JSON
      - Logs sign-in attempt to audit_log table (action: 'staff_sign_in', target_id: staff_id)
      - 401 on bad credentials with generic message (no user enumeration)
      - 429 not implemented this batch (rate limiting deferred)
    - POST /v1/auth/pin:
      - Validates body via PinRequestSchema
      - Looks up staff_members where tenant_id, location_id matches AND role = 'cashier'
      - For each candidate, bcrypt-verifies PIN against staff.pin_hash
      - On match: issue terminal JWT
      - Logs to audit_log (action: 'cashier_pin_sign_in')
      - 401 on no match
      - Constant-time-ish: always iterate full candidate list, never short-circuit, to limit timing attacks (acceptable for MVP since cashier set is small per location)

15. Mount auth router in apps/pos-api/src/index.ts at /v1/auth.
