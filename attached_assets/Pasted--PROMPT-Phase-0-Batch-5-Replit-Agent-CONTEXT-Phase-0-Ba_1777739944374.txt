   - MenuCategorySchema = z.object({ id: z.string().uuid(), tenant_id: z.string().uuid(), name: z.string().min(1).max(100), sort_order: z.number().int().nonnegative(), deleted_at: z.string().datetime().nullable(), created_at: z.string().datetime() })
   - MenuItemSchema = z.object({ id, tenant_id, category_id: z.string().uuid(), name: z.string().min(1).max(120), price_cents: z.number().int().nonnegative().max(1000000), taxable: z.boolean(), image_url: z.string().url().nullable(), kitchen_station: z.string().nullable(), deleted_at, created_at })

SECTION 2 — pos-api menu routes
═══════════════════════════════════════

5. apps/pos-api/src/lib/db.ts (new helper, or add to existing supabase.ts):
   - Helper functions for tenant-scoped queries that always inject tenant_id from req.auth
   - Example: tenantQuery(req, 'menu_items') returns a query builder pre-filtered by tenant_id
   - This prevents accidental cross-tenant reads even though service_role bypasses RLS

6. apps/pos-api/src/middleware/role-guard.ts:
   - requireRole(['owner', 'manager']) — middleware that 403s if req.auth.role not in allowed list
   - For terminal JWT (kind: 'terminal'), role is always 'cashier', so this naturally restricts terminal sessions to read-only menu routes

7. apps/pos-api/src/routes/menu/categories.ts:
   - GET /v1/menu/categories — any auth (session OR terminal); returns active (deleted_at IS NULL) categories for tenant, sorted by sort_order
   - POST /v1/menu/categories — owner+manager only; body = CreateMenuCategoryRequest; inserts with tenant_id from JWT, returns created row
   - PATCH /v1/menu/categories/:id — owner+manager; tenant-scoped UPDATE; returns updated row; 404 if not found in tenant
   - DELETE /v1/menu/categories/:id — owner+manager; soft delete (set deleted_at); returns 204

8. apps/pos-api/src/routes/menu/items.ts:
   - GET /v1/menu/items — any auth; query params: category_id (optional filter), include_deleted (owner+manager only); returns array
   - POST /v1/menu/items — owner+manager; validates category_id belongs to tenant before insert; returns created
   - PATCH /v1/menu/items/:id — owner+manager
   - DELETE /v1/menu/items/:id — owner+manager; soft delete

9. apps/pos-api/src/routes/menu/modifier-groups.ts:
   - GET /v1/menu/modifier-groups — any auth; returns groups + their options (one query with join, returned nested)
   - POST /v1/menu/modifier-groups — owner+manager
   - PATCH /v1/menu/modifier-groups/:id — owner+manager
   - DELETE /v1/menu/modifier-groups/:id — owner+manager; soft delete

10. apps/pos-api/src/routes/menu/modifier-options.ts:
    - POST /v1/menu/modifier-options — owner+manager; body includes group_id; verify group belongs to tenant before insert
    - PATCH /v1/menu/modifier-options/:id — owner+manager
    - DELETE /v1/menu/modifier-options/:id — owner+manager; HARD delete (no soft delete on options — they're cheap to recreate, no historical orders reference them directly since modifiers are stored as JSON snapshots on order_items)

11. apps/pos-api/src/routes/menu/links.ts:
    - POST /v1/menu/items/:item_id/modifier-groups — owner+manager; body: { group_id, sort_order }; inserts into menu_item_modifier_groups; verify both belong to tenant
    - DELETE /v1/menu/items/:item_id/modifier-groups/:group_id — owner+manager

12. apps/pos-api/src/routes/menu/tree.ts:
    - GET /v1/menu/tree — any auth; returns full nested menu tree (categories → items → modifier groups → options) for the cashier app to consume in one call
    - Single endpoint that powers the terminal's offline-cache priming
    - Excludes deleted items
    - Response shape matches MenuTreeResponseSchema

13. Mount all menu routers under /v1/menu in apps/pos-api/src/index.ts.

14. Every route uses the requireAuth middleware (from Batch 4) plus, where applicable, requireRole.

15. Every mutation logs to audit_log: action like 'menu_item_created', target_type 'menu_item', target_id, payload: snapshot of the change.
