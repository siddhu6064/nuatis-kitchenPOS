CONTEXT
Phase 0 Batches 1-11 complete: 12 migrations, full API (auth, menu, orders, KDS, cash, receipts), 76 tests passing. This batch closes Phase 7 of the build checklist: end-of-day reports. Architecture: snapshot-based — a BullMQ cron worker runs at 00:01 tenant-local time, computes daily totals, writes to reports_daily table; the API reads from the snapshot for historical days, falls back to live aggregation for "today." This gives owners immutable historical numbers even when refunds/voids hit later.

GOAL
Working report pipeline: GET /v1/reports/end-of-day?date=YYYY-MM-DD returns gross_sales / taxable / tax / tips / discounts / voids / refunds / net + by-item / by-staff / by-method breakdowns. Cron worker snapshots yesterday's numbers at midnight tenant-local. Optional email-to-owner via existing Resend wiring when owner has email_daily_report=true on tenant. CSV export endpoint. NO admin UI this batch — that's Batch C.

SECTION 3 — pos-api: aggregation library
═══════════════════════════════════════

3. apps/pos-api/src/lib/reports.ts — pure aggregation logic:
   - Pure functions, no Express, no DB client smuggled in
   - aggregateEndOfDay(orders, orderItems, payments, refunds, cashEvents, staffMembers, menuItems, salesTaxBps) → returns EndOfDayReportSchema-shaped object minus tenant/location/date/is_snapshot fields
   - This is unit-testable without DB; tests pass deterministic fixtures
   - Aggregation rules:
     * gross_sales_cents = sum(payments.amount_cents - payments.tip_cents) for status=succeeded
     * tips_cents = sum(payments.tip_cents) for status=succeeded
     * tax_cents = sum(orders.tax_cents) for status=paid
     * taxable_cents = sum(order_items.price_cents * quantity) for non-voided items where menu_item.taxable=true
     * voids_cents = sum(orders.subtotal_cents + tax_cents) for orders.status=voided ON THE DATE (use voided_at, not opened_at)
     * refunds_cents = sum(refunds.amount_cents) where created_at::date = report_date
     * discounts_cents = 0 for MVP (no discount feature yet — placeholder for future)
     * net_cents = gross_sales_cents + tips_cents - refunds_cents
     * by_method: GROUP BY payments.method, count, sum(amount_cents - tip_cents)
     * by_item: GROUP BY order_items.menu_item_id (only non-voided), sum quantity, sum price_cents*qty, compute pct_of_total
     * by_staff: GROUP BY orders.opened_by_staff_id, count distinct order_id, sum gross, sum tip
   - Date boundary: a row counts for date X if its created_at falls within X tenant-local 00:00:00 to 23:59:59. Pass tenant timezone as a parameter.
   - All math in integer cents; never decimal

4. apps/pos-api/src/lib/reports.test.ts — extensive unit tests:
   - Empty-day: no orders → all zeros
   - Single cash sale, no tip → gross_sales=400, tips=0, by_method=[{method:'cash',count:1,gross:400}]
   - Multiple sales different methods → by_method aggregates correctly
   - Voided order excluded from gross
   - Voided order counted in voids_cents and voided_order_count
   - Refund on the same day reduces net
   - Tip calculation correct
   - by_item percentages sum to 100% (within 0.5% rounding tolerance)
   - by_staff aggregation correct
   - Items where menu_item is null (e.g., deleted menu item) handled gracefully
   - Timezone boundary: order created 23:55 of date X stays in date X; 00:05 next day goes to next day
   - At least 12 unit tests in this file

   - GET /v1/reports/end-of-day — session-only (owner+manager); requireRole(['owner', 'manager'])
   - Query params validated by GetEndOfDayQuerySchema
   - Logic:
     * If date == today (tenant-local): run live aggregation, return is_snapshot=false
     * If date < today: try reports_daily lookup; if exists return is_snapshot=true; if not, run live aggregation, return is_snapshot=false (snapshot will populate when next cron fires)

   - Pass tenant timezone from tenant row (default 'America/Chicago' from foundation)
   - Returns EndOfDayReportSchema-shaped JSON

   - Format: standard daily summary on top rows, blank, then by_item rows, blank, then by_staff rows, blank, then by_method rows
   - Use csv-stringify@^6 (mature, no nonsense) — add to deps
   - All money formatted as decimal dollars in CSV (e.g., "12.50") for spreadsheet readability — convert from cents at the boundary
