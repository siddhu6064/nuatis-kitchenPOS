▶ PROMPT [Phase 2 · Batch 4 · Replit Agent]

CONTEXT
Batches 1–3 shipped: auth, menu grid, cart with controls, mock checkout flow
ending at an Approved screen. Order data persists to localStorage under
`nuatis_pos_last_order_v1`. The "View Receipt" button on the Approved screen
is currently a console.log stub. Now: build the receipt screen UI.
Repo: github.com/siddhu6064/nuatis-kitchenPOS.

GOAL
After tapping "View Receipt" on the Approved screen, cashier sees a clean
printable receipt with full order detail, can return to a new order, and
the layout reads correctly on both tablet and printed letter-sized paper.

SCOPE — BUILD THIS
1. Receipt route / view — fullscreen, white background, dark text, narrow
     centered column (~360px wide) so it reads like a real thermal-printer receipt
2. Read order data from localStorage key `nuatis_pos_last_order_v1`. If empty
     or missing, show "No recent order" state with a back button to menu
3. Receipt header — "Nuatis POS" wordmark (or text), mock cafe name
     "Demo Cafe", mock address "123 Main St, Austin, TX 78701", mock phone
     "(512) 555-0100"
4. Order metadata — order ID (generate as `ORD-` + last 6 chars of timestamp),
     date/time formatted human-readable (e.g. "May 2, 2026 · 2:47 PM"),
     "Card · Approved" payment line
5. Line items section — each item: qty × name on left, line total on right.
     Monospace or tabular figures so columns align cleanly
6. Totals block — subtotal, tax (8.25%), tip, divider, grand total in bold
7. Footer — "Thank you" message, mock "Powered by Nuatis" line, dashed
     divider rule above and below the items section
8. Action bar (NOT printed — hidden via `@media print`):
     - "Print" button → triggers `window.print()`
     - "Email" button → stub, opens an alert "Email receipt — coming soon"
     - "Back to Approved" secondary button → returns to approved screen
     - "New Order" primary button → clears cart + last_order, returns to menu
9. Print stylesheet — when printing, hide app chrome (header, action bar,
     navigation), show only the receipt column, remove background, ensure
     receipt fits on standard letter and 80mm thermal width
10. Wire the existing "View Receipt" button on the Approved screen to
     navigate to this receipt view (replace the console.log stub)
11. Receipt is also accessible after navigating away — if user lands on
     receipt route directly, it reads from localStorage and renders the last
     completed order

SCOPE — DO NOT BUILD
- Real email or SMS sending
- Receipt history list (only most recent order this batch)
- QR code, loyalty marketing, customer info capture
- PDF export
- Multi-receipt formats / template editor

TECH CONSTRAINTS
- React + Vite, Tailwind v3, lucide-react (Printer, Mail, ArrowLeft icons)
- Tabular figures via `font-variant-numeric: tabular-nums` for amount columns
- Print stylesheet via Tailwind `print:` modifiers OR a single `@media print`
     block in index.css — whichever is cleaner
- Keep cents-safe math pattern from Batch 3 — no float drift in totals
- All buttons ≥44×44px, ARIA labels on icon-only buttons

ACCEPTANCE
- Tapping "View Receipt" on Approved screen navigates to receipt view
- Receipt renders all line items, totals, tip, payment method correctly
     pulled from localStorage
- Order ID format `ORD-XXXXXX`, date/time human-readable
- Print preview (Cmd/Ctrl+P or Print button) shows ONLY the receipt column —
     no app chrome, no action buttons, no background color
- Receipt fits on letter paper without overflow; reads cleanly at thermal
     80mm width
- Empty state renders correctly when localStorage has no order
- "New Order" clears cart + last_order and returns to menu shell
- "Back to Approved" returns user to the approved screen with state intact
- Direct navigation to receipt route works after a refresh (reads localStorage)
