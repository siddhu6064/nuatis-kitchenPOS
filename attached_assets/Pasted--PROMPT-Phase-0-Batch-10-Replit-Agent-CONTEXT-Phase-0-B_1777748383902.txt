   - GET /v1/cash/sessions — session-only (owner+manager); query params: location_id (optional), status (optional), limit (default 50); returns array

SECTION 4 — Manager PIN override pattern
═══════════════════════════════════════

7. Create apps/pos-api/src/middleware/manager-pin.ts:
   - Middleware factory: requireManagerPin() — checks if request body includes a manager_pin field
   - If absent: returns 403 with body { error: { code: 'manager_pin_required', message: 'A manager PIN is required for this action.' } }
   - If present: looks up staff_members where tenant_id matches AND role IN ('owner', 'manager'), iterates all to bcrypt-compare manager_pin against pin_hash (constant-ish time)
   - On match: attaches req.manager_id to request, calls next()
   - On no match: returns 403 with code 'manager_pin_invalid'
   - Document in code comments: client UX is "user attempts action → 403 manager_pin_required → modal pops up asking manager to enter PIN → client retries same request with manager_pin in body"

8. Apply requireManagerPin to:
   - POST /v1/cash/sessions/:id/events when type='pay_out' or type='no_sale'
   - POST /v1/orders/:id/void (existing endpoint — currently session-only; add manager_pin override path so cashier can void with manager approval)
   - Future: POST /v1/payments/:id/refund (Stripe batch — not this batch)
   - Future: discount endpoints (not built yet)

9. Audit log entries on every manager-overridden action: action='manager_pin_override', payload includes manager_id and the original action.
